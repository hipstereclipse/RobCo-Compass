A practical hand-off guide. If you've just inherited this repo and want to make a

that you know Espruino or the Pip-Boy.

## 1. Mental model in 60 seconds

- **You edit exactly one file:** [`src/COMPASS.JS`](../src/COMPASS.JS) — readable,
  commented ES5.
- **The device runs a *minified* copy:** `APPS/COMPASS.JS`, generated by the
  build. **Never hand-edit the minified file.**
- **You install by copying files to the SD card** (or with the web installer).
  **You never "flash" or "Send to Espruino"** — that risks the firmware.
- **Three checks before you ship:** `npm test` (logic), `npm run shots` (visual),
  and a real-device/simulator pass.

---

## 2. Set up your toolchain

You need **Node.js 16+** (only for the dev tools — the app itself ships no Node
dependency) and, for the one-click installer, **Chrome or Edge**.

```bash
git clone https://github.com/hipstereclipse/RobCo-Compass.git
cd RobCo-Compass
npm install        # installs terser (the only devDependency)
npm test           # should print: RESULT: ALL PASS
```

If `npm test` passes on a clean clone, your environment is good.

## 4. The four pipelines

### Build (`npm run build`)
[`tools/build.js`](../tools/build.js) minifies the source with **terser** and
writes `APPS/COMPASS.JS` (~13.5 KB vs ~35 KB source — the kind of reduction that
keeps you clear of `LOW_MEMORY`). If terser isn't installed it falls back to a
conservative comment/whitespace strip and warns. The build sets
`compress.side_effects:false` for a critical reason — see [Footguns](#7-footguns--gotchas).

### Icon (`npm run icon`)
[`tools/make-icon.js`](../tools/make-icon.js) draws a 64×64 compass glyph and packs
it into the Espruino 1-bpp image format (`width, height, bpp`, then MSB-first
bits) the launcher expects.

### Screenshots (`npm run shots`) — the clever one
[`tools/render-screens.cjs`](../tools/render-screens.cjs) is **not** a mock-up. It:

1. reads the real `src/COMPASS.JS`,
2. evaluates the app factory against a **software rasterizer** that implements the
   exact `Graphics` calls the app makes (`setColor`, `drawString`, `drawLine`,
   `drawRect`, `fillRect`, `drawCircle`, `setFontAlign`, `stringWidth`),
3. drives the app into each screen state through its **real** knob/torch handlers
   (exactly like the test harness),
4. captures the framebuffer, applies a phosphor-CRT post-process (bloom +
   scanlines + vignette), and encodes a PNG with `zlib` — **zero dependencies**.

Because the shipping code path draws the images, the documentation can't silently
drift from the UI. Change the UI, run `npm run shots`, commit the updated PNGs.

### Test (`npm test`)
[`tools/test-harness.cjs`](../tools/test-harness.cjs) stubs the Espruino globals,
evaluates the factory, drives the full flow and asserts: the honest `MODE` line,
solar capture + persistence, synchronised Morse flashing, the sensor-upgrade
branches (magnetometer → MAGNETIC, gyro → DEAD-RECKONING), and that `remove()`
detaches every listener/timer and leaves the torch off. It's a **logic** test, not
a hardware substitute.

## 5. The everyday change loop

```bash
# 1. edit the source
$EDITOR src/COMPASS.JS

# 2. prove the logic still holds
npm test

# 3. if you changed anything visible, refresh the screenshots and eyeball them
npm run shots

# 4. produce the device build
npm run build

# 5. (icon only if you changed make-icon.js)
npm run icon

# 6. verify for real — simulator first, then device REPL (see §8)

# 7. install by COPYING APPS/COMPASS.JS + APPINFO/* to the SD card
#    (or serve install.html and use the one-click installer)
```

Commit `src/`, the regenerated `APPS/COMPASS.JS`, and any changed
`screenshots/` / `APPINFO/HOLO.IMG` together so the repo stays internally
consistent.

## 6. Recipes

### Recipe: add a soft button / new screen on HOME
1. Add a `SCR.MYSCREEN` id ([routing block](../src/COMPASS.JS#L434-L435)).
2. Add a button entry in `drawHome()`'s `btns` array and bump the `homeFocus`
   modulus in `onKnob1` so the highlight can reach it.
3. Add a `gotoMyScreen()` and route to it from `press1()`.
4. Write a `drawMyScreen()` and add a branch to the `draw()` dispatcher.
5. Handle its input in `onKnob1`/`onKnob2`/`press1`/`onTorch` (give TORCH a
   "back to HOME" path).
6. `npm test` → add an assertion → `npm run shots` to capture it.

### Recipe: add a field to CALIBRATE
1. Add the field name to `CAL_FIELDS` ([source](../src/COMPASS.JS#L442)).
2. Add its display string to the `vals` map in `drawCalibrate()`.
3. Add a `changeField()` branch for how K1 edits it
   ([source](../src/COMPASS.JS#L700-L710)).
4. If it should persist, add the key to `cfg` ([source](../src/COMPASS.JS#L94-L106))
   so `loadCfg`/`saveCfg` carry it automatically.

### Recipe: add a Morse keypad key
1. Add the label to the relevant row in `KEYS`
   ([source](../src/COMPASS.JS#L451-L457)).
2. Handle it in `morseActivate()` ([source](../src/COMPASS.JS#L731-L745)).
   Letters/digits already fall through to "append to message".

### Recipe: light up a magnetometer (when hardware/firmware gains one)
The MAGNETIC strategy is already written and tested — you mostly need to teach the
probe how to *find* the sensor on the new firmware:
1. In `Heading.probe()` ([source](../src/COMPASS.JS#L233-L245)) confirm the names
   `Pip.magRd / getMagnetometer / mag` match what the firmware exposes; add the
   real one if it differs.
2. Do the same for `readMag()` ([source](../src/COMPASS.JS#L247-L252)).
3. Verify on device that `magHeading()`'s tilt-compensation gives a sane reading,
   then calibrate hard/soft-iron offsets in `magCal`.
4. `npm test` already asserts the branch selects `MAGNETIC` when `magRd` exists.

### Recipe: add or change a theme colour
Add an entry to `THEMES` ([source](../src/COMPASS.JS#L54-L57)) as
`[r, g, b]` fractions of a single hue, and extend the toggle in `calActivate()`'s
`TOGGLE THEME` action. Keep the **one-hue** rule: the four `LEVEL` brightnesses are
your only on-screen variation.

### Recipe: regenerate the screenshots
```bash
npm run shots
```
Writes `screenshots/01-home.png … 05-home-amber.png` and
`preview-contact-sheet.png`. To add a new screen to the set, add a scenario to the
`SHOTS` array in [`tools/render-screens.cjs`](../tools/render-screens.cjs) with a
small `drive(ctx)` function that walks the app there using `ctx.knob1/knob2/torch/
tick` (mirror the existing drivers like `driveCalibrate`).

## 7. Footguns & gotchas

These are the things that will bite you specifically on this platform.

- **ES5 only.** No `let`/`const`/arrow functions/template strings in
  `src/COMPASS.JS`. Espruino is ES5. (The *dev tools* in `tools/` run under Node
  and may use modern JS — that's fine, they never ship.)
- **The app is a function *expression* with no trailing `()`.** The loader calls
  it. Don't "fix" it into an IIFE.
- **terser `compress.side_effects:false` is required.** Without it the minifier
  deletes the top-level function expression as a "useless statement" and you ship
  an **empty** file. `tools/build.js` already sets it — don't remove it.
- **`remove()` must be airtight.** Every timer cleared, every `Pip.on` listener
  detached, torch/LED forced off, config flushed. A leak fires into a dead app and
  corrupts the next one. The test asserts this — keep it green.
- **Feature-detect every hardware touch.** Graphics instance, accel, mag/gyro,
  torch, audio, storage — wrap them and degrade. Never assume an API exists.
- **K2 has no press** on this firmware. Never design an interaction that requires
  one.
- **`setFontAlign` is `-1/0/1`,** not `0/0.5/1`.
- **One hue on screen, ever.** Brightness levels only.
- **Never flash / "Send to Espruino".** Install by copying files. Back up the SD
  card first.

## 8. Verifying on real hardware

`npm test` proves the *logic*; it cannot prove the *hardware API* on your
firmware. Always do a real pass:

1. **Simulator first** — prototype on the [pip-boy.com](https://www.pip-boy.com)
   3000 simulator.
2. **Device REPL** — in the Espruino Web IDE over Web Serial (Chrome), confirm
   what your firmware actually exposes before relying on it:
   ```js
   Object.keys(Pip).filter(k => /accel|mag|gyro|torch|led/i.test(k));
   Pip.on('accel', a => print(a));   // is there an accel event stream?
   ```
3. **Install by copying** `APPS/COMPASS.JS` + `APPINFO/*` to the card and reboot;
   it appears under **ITEMS › MISC** as **Compass**.
4. Re-confirm the `remove()` teardown by entering and exiting the app a few times
   and checking the torch/LED is off and the next app behaves.

## 9. Glossary

| Term | Meaning |
|---|---|
| **Lubber line** | The fixed index at the top of the rose marking "the direction the device points." |
| **Bearing** (held) | In MANUAL mode, the heading you've told the app you're facing; the rose is drawn relative to it. |
| **Trim** | A small constant offset added to every reading, for fine correction. |
| **Declination** | The angle between magnetic north and true north; only meaningful in MAGNETIC mode. |
| **Azimuth** | A compass direction in degrees clockwise from true north (the sun's azimuth is what `CAPTURE SUN` uses). |
| **EMA** | Exponential moving average — the smoothing that makes the rose glide instead of jump. |
| **Dead reckoning** | Estimating heading by integrating turn-rate from a known start; drifts over time. |
| **Hard/soft iron** | Magnetometer calibration offsets (`magCal`) correcting for nearby metal and sensor scale. |
| **WPM / unit** | Morse speed; one time unit = `1200 / WPM` ms (PARIS standard). |
| **`flip()`** | Pushes the offscreen buffer to the panel, producing the native scanline/phosphor look. |
